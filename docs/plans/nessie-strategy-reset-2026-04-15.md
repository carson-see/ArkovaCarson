# Nessie Strategy Reset — 2026-04-15

> **Status:** Diagnosis + plan. No code training resumes until infrastructure passes the smoke test in §6.
> **Author:** generated during the v26 deploy attempt that surfaced the cascade of latent failures below.
> **Supersedes:** the implicit "train more, eval later" loop that produced v6–v26.

## TL;DR

Six months of Nessie work has produced **zero deployed inference**. The v2 baseline showing "0% F1" was not a model-quality problem — it was an infrastructure problem: every extraction call timed out against a dead RunPod endpoint, and every Together fine-tune is silently non-serverless (requires a paid dedicated endpoint). Production has been quietly running base Gemini 2.0 Flash for everything.

This doc resets the strategy: stop training, fix infrastructure, prove ONE domain end-to-end before expanding.

## 2. Root cause: why we got here

1. **Training-first loop with no deployment proof.** v6 → v7 → … → v26 each added domains, each was claimed done, but no version was verified to serve a single inference request end-to-end. Jira closures are not deployment proof.
2. **Catastrophic forgetting compounded over 21 LoRAs.** Each iteration trained on top of the previous, never going back to an eval-anchored checkpoint. Even if the final v26 *could* serve, it has likely lost most of the format-correctness from v3–v5.
3. **Infrastructure assumed, not tested.** The eval runner's `nessie` provider hits a `RUNPOD_ENDPOINT_ID` env var — and nobody verified that endpoint still existed. Together's serverless requirement was never re-checked after Together changed its policy.
4. **Generalist sprawl.** v17 (NY privacy), v18 (CA privacy), v19 (HIPAA deep), v20 (SOX), v21 (FERPA), v22 (FCRA), v23 (Kenya), v24 (Australia), v25 (GDPR), v26 (Africa) — 12+ regulatory domains crammed into one 8B model with ~10MB of data per version. Best practice is the opposite: master narrow before expanding.
5. **Local disk treated as model registry.** Every "deploy" attempt went `Together → local merge → HF upload → RunPod`. With local disk at 99% full and merging 12GB models, this path was always going to break.

## 3. Best-practices research (what real production fine-tunes look like)

These are the consensus patterns from production NLP teams (Anthropic alignment work, OpenAI fine-tuning docs, Llama-3 community, the recent Gemini distillation papers, and what Together themselves recommend):

### 3.1 Format-correctness BEFORE knowledge

Before teaching a model new domain knowledge, prove it outputs valid JSON 100% of the time on simple inputs. Use **constrained decoding** (vLLM grammar, JSON Schema mode, or Together's `response_format`) — don't rely on the model to format correctly.

> **Why this matters here:** the v2 baseline reported "0% F1" partly because Nessie was not even returning parseable JSON. Even when the endpoint works, the model is being asked to produce structured output without any decoding constraints. Add JSON-Schema constrained decoding before any further training.

### 3.2 Quality > quantity for SFT

Anthropic's RLHF papers and the Llama-2 / Llama-3 SFT ablations consistently show that **100 hand-curated, diverse, high-quality examples outperform 10,000 auto-generated ones**. The 4,617 examples in v26 were generated by a script — there's no quality gate.

> **Application:** for our first single-domain run, 50–100 hand-labeled DEGREE credentials beats anything we can generate.

### 3.3 Eval-driven iteration, never train without a hypothesis

Every fine-tune should be motivated by a specific failure mode in the prior version's eval. "Training more on more domains" is not a hypothesis. "v5 misses `creditHours` 60% of the time on California CLE certificates with PDF tables — train on 30 hand-labeled examples of those" is a hypothesis.

> **Rule:** no training run starts without (a) a named failure mode, (b) a measurable target, (c) a proof of where it will deploy.

### 3.4 Stop continuous LoRA stacking

LoRA fine-tunes do not compose cleanly. Each new fine-tune is trained against the previous merged weights. Catastrophic forgetting means by v26 we likely cannot reliably extract things v5 used to handle.

> **Rule:** future fine-tunes start from the **base instruct model + a clean recipe**, not from the previous version. Treat each version as fresh.

### 3.5 Single-domain mastery before expansion

Pick one credential type, get it to ≥85% F1 with calibrated confidence (Pearson r ≥ 0.7), deploy it, watch it run in production for a week, then add the next type. This was the user's stated direction and it aligns with what every production team does (e.g., Stripe's Radar started with one fraud type).

### 3.6 Distillation often beats SFT for extraction

Use Gemini Pro / Claude as a teacher — generate 1k-5k high-quality `(input, structured_output)` pairs, distill into the smaller model. This is what most production pipelines actually use; pure SFT on hand-labels is rarely worth the effort below ~10k examples.

### 3.7 Confidence is a separate problem

Confidence calibration cannot be fixed by training the model harder. Either train a separate calibration head, use temperature scaling on logits, or use ensemble agreement. Our current "confidence" is the model's self-report, which is well-known to be noisy.

## 4. Strategy reset: rules going forward

1. **Dual-track stays.** Together for training (cheap LoRA fine-tuning). RunPod for serving (cost-effective serverless inference). Local is **never** for model artifacts.
2. **No new training until v26 (or a chosen base model) is deployed end-to-end and beats Gemini base on the same eval set.** No exceptions.
3. **One domain at a time.** Order: DEGREE → CLE → PROFESSIONAL → LICENSE → CERTIFICATE → … (highest-volume first per golden dataset distribution).
4. **Per-domain Definition of Done:**
   - 50+ hand-labeled examples for the domain
   - F1 ≥ 85% on a held-out 20-sample test set for that domain
   - Confidence Pearson r ≥ 0.7 on that domain
   - Deployed to RunPod, healthy endpoint, latency p95 < 5s warm
   - Beats `gemini-2.0-flash` base on the same 20-sample test set, OR matches it at <50% the cost
   - One full week of production traffic without a circuit-breaker trip
5. **Eval is the gate, not Jira.** A story is "Done" when the eval doc shows the DoD met. Until then it stays open regardless of what the code says.
6. **No model artifacts on local disk, ever.** All merges, downloads, and HF pushes happen on RunPod GPU pods (described in §7).
7. **Archive don't delete.** Old fine-tunes (v6–v26) stay on Together for forensic comparison but are removed from any default config.

## 5. What to throw out

- The v6–v26 generalist sprawl. Mark as "training experiments, not deployment candidates" in CLAUDE.md.
- The "12+ regulatory domains in one 8B model" approach. Domains will be served by separate adapters (per existing domain-router design) or by routing to Gemini for low-volume types.
- The local-merge → local-HF-push deploy path. Replaced by §7.
- The "Jira Done = production-ready" assumption. The Nessie Production Hardening section in CLAUDE.md already calls this out — make it a hard gate.

## 6. Smoke test — what must pass before resuming training

A single `make nessie-smoke` target that, in order:

1. Confirms `RUNPOD_ENDPOINT_ID` resolves to a live endpoint via `runpod GraphQL myself { endpoints }`.
2. Confirms `NESSIE_MODEL` is hosted at the expected URL (HF API call returns >5 files).
3. Sends one PII-stripped extraction request through the worker → gets back a parseable JSON object matching `ExtractedFieldsSchema` → confidence in [0,1].
4. Compares against `gemini-2.0-flash` base on the same 5 inputs — Nessie must not be worse on >2/5.

**No new training, no new domain work, until this passes.**

## 7. RunPod-native upload pipeline (replaces local-merge path)

A one-off RunPod pod (8x A100 or single A100 80GB, ~$2-4 per run) that:

1. `together cli download <model-id>` — pulls LoRA from Together (free, fast on RunPod's network)
2. `python merge_lora.py` — merges into `meta-llama/Meta-Llama-3.1-8B-Instruct` (held in pod's persistent volume)
3. `huggingface-cli upload carsonarkova/<model-name> ./merged --commit-message "v<n> merged"` — pushes to HF (also fast on RunPod's bandwidth)
4. Updates the RunPod serverless template's `MODEL_NAME` to the new HF path
5. Pod terminates

Encapsulate as `services/worker/scripts/runpod-deploy-nessie.sh` invoking a RunPod template ID. Total time: ~20 minutes per deploy. Local disk used: zero.

## 8. Single-domain plan: DEGREE first

**Why DEGREE?** Highest entry count in the golden dataset (~700 across phases), most uniform format (institution + degree level + field of study + date), lowest fraud signal complexity, highest production volume from public records ingest.

### Phase 1: Curate (1 day)
- Pull 80 DEGREE entries from existing golden dataset
- Manually correct any mislabeled fields (script-generated examples often have inconsistent date formats, missing accreditingBody, etc.)
- Hold out 20 as test set, never train on these

### Phase 2: Train (4 hours, $30 on Together)
- Base: `meta-llama/Meta-Llama-3.1-8B-Instruct-Reference` (clean, no prior fine-tune)
- Training data: 60 DEGREE-only examples, JSON-formatted output
- LoRA rank 16, 3 epochs, lr 1e-4
- Save as `arkova-nessie-degree-v1`

### Phase 3: Deploy via §7 pipeline (20 min, $4 on RunPod)
- Merge + push to HF
- Update RunPod serverless template
- Update worker `.env` `NESSIE_MODEL`

### Phase 4: Eval (1 hour)
- Run `npx tsx run-eval.ts --provider nessie --sample 20` on held-out DEGREE set
- Compare against Gemini base (`--provider gemini --sample 20` filtered to DEGREE)
- Document in `docs/eval/eval-nessie-degree-v1-<date>.md`

### Phase 5: Production canary (1 week)
- Update Cloud Run worker: enable hybrid routing for DEGREE only
- Monitor circuit breaker, latency p95, fallback rate
- If all metrics green for 7 days, mark domain done. Move to CLE.
